Listes html

Eléments gérant une liste

  • Une liste est traitée en html comme une suite d'éléments
    C'est donc un élément list regroupant des items (ses fils directs)

  • Une list est toujours ordonnée

    • Si cet ordre doit être explicité à l'utilisateur final, le tag utilisé sera ol

    • Sinon, le tag utilisé sera ul

  • Les item portent toujours le tag li

  • On obtient les deux syntaxes suivantes 

    <ol>
        <li> ... </li>
            ...
        <li> ... </li>
    </ol>

    <ul>
        <li> ... </li>
            ...
        <li> ... </li>
    </ul>

  • Une grande variété de propriétés contrôle la mise en page : doc   conseils

Tableaux html

Structure des tableaux en html

  • Un tableau est un élément html spécialisé de tag table

  • Ses fils doivent apparaitre dans un ordre imposé :

    1. caption : (optionnel) usuellement un texte décrivant le contenu tableau

    2. colgroup : (optionnel) description de regroupement de colonnes (peu utilisé)

    3. thead : (optionnel) lignes réservées à la structuration des noms de colonnes

    4. tbody : lignes de contenu

    5. tfoot : (optionnel) lignes réservées à l'affichage de résumés sur le contenu

  • Les lignes de thead, tbody, ou tfoot, sont aussi des éléments structurés

    • Chaque ligne est un élément portant le tag tr (table row)

    • Elle doit avoir autant de cases, i.e. de fils td (table data) que nécessaire
      pour être cohérente avec le nombre de colonnes

    • Les cases peuvent aussi être étiquetées th, pour distinguer celles qui jouent un rôle particulier d'entête (héritage html4 ; parfois redondant avec thead)

Propriétés css pour les tableaux

  • La propriété caption-side permet de positionner l'élément caption

  • Le pseudo sélecteur :nth-child() permet d'alterner
    entre les lignes paires et impaires, entre autres

  • La sélection par attribut permet de cibler
    les fusions de lignes/colonnes : [rowspan] / [colspan]

  • Au sein de chaque groupe sémantique, la distinction td / th
    repère la distinction entre cases de contenu / de nommage

  • Les sélecteurs composés prennent tout leur sens
    pour différencier les lignes de thead, tbody et tfoot

Formulaires html

Formulaires Web

  • Les formulaires jouent un rôle particulier dans le modèle web

  • Il permet à l'utilisateur final d'envoyer des informations au serveur

  • Celui-ci répondra en renvoyant un nouveau document html

  • Dans ce cours nous ne nous occupons pas du côté serveur

  • Nous allons seulement voir comment construire les formulaires côté client,
    c'est-à-dire dans le document html, pour le présenter à l'utilisateur final

Structure de l'élément formulaire : racine form

  • Le formulaire est un élément html spécialisé : form

  • Les éléments du formulaire, i.e les fils de form, n'ont pas d'ordre imposé

  • L'élément form gère la communication avec le serveur via deux attributs :

    • action : url (côté serveur) qui recevra l'envoi des informations saisies par l'utilisateur

    • method : méthode HTTP utilisée pour la transmission des données : GET ou POST

  • Remarques :

    • GET ne peut être employé que pour des petits volumes de données (<3Ko)

    • En particulier pour l'envoi de fichiers, il faut utiliser POST

    • avec POST, un troisième attribut est requis :
      enctype avec la valeur multipart/form-data

  • (hors sujet) exploitation en local
    → method="dialog" : aucun envoi / traitement en script

Structure de l'élément formulaire : paires label/input

  • Les champs de saisie s'organisent via une paire d'éléments :

    • label : description de la saisie est attendue, destinée à l'utilisateur final

    • input : élément pour la saisie effective, spécialisé via l'attribut type

  • Chaque input porte son id (unique)
    Le label lui est associé en le recopiant dans son attribut for

  • Chaque input doit avoir un attribut name pour permettre la transmission
    Sa valeur servira de clé pour les informations collectées par l'input

  • Quelques autres attributs courants des input

    • required : doit être renseigné pour autoriser l'envoi

    • placeholder : permet d'afficher un indice sur ce qui est attendu

    • value : permet de préremplir

Structure de l'élément formulaire : submit et reset

  • Un élément input de type="submit" permet de déclencher l'envoi :
    <input type="submit" value="message affiché" />

  • Il prend la forme d'un bouton que l'on retrouvera a priori dans tout les formulaires, à la fin

  • Couramment, il lui est associé un input prenant aussi la forme d'un bouton
    qui permet la remise à zéro du formulaire :
    <input type="reset" value="message affiché" />
    Il est bien entendu optionnel

L'input couteau suisse : type="text"

  • Par défaut, on peut toujours se rabattre sur le type le plus simple
    <input type="text"/>

  • Celui-ci permet à l'utilisateur de saisir une string

  • Les attributs classiques de taille sont disponibles : min et max

  • S'agissant de string, tout informaticien pense "expressions régulières"
    L'attribut pattern permet de définir une expression régulière qui servira à contrôler le genre de string saisie

  • L'attribut list permet d'utiliser une liste de complétion
    Sa valeur est l'id d'un élément datalist, dont les fils (option) énumèrent les choix possibles via leur attribut value
    (la datalist peut être définie n'importe où dans le body, et ne sera jamais affichée)

  • Toute autre contrôle/vérification/traitement est responsabilité du serveur

Utiliser du css pour un formulaire

  • Dans le cas des formulaires, certains aspects de la mise en page sont de notre responsabilité, mais d'autres resteront entre les mains du navigateur (sauf à faire de très gros efforts …)

  • L'ergonomie est cruciale afin de permettre une saisie efficace
    Toutes les erreurs de saisie nous pénalisent lors du traitement

  • Une erreur détectée au niveau du serveur implique
    de gérer des aller/retour et des répétitions de saisie

  • En plus de ce que nous connaissons, css ajoute deux pseudo-classes
    :valid et :invalid

  • Attention, pour cela, il faut être capable de définir ce que c'est qu'une information correcte, sans avoir recours aux informations stockées seulement sur le serveur
